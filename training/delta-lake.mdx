Delta Lake는 오브젝트 스토리지 위에 신뢰할 수 있는 데이터 레이어를 제공하는 오픈 소스 스토리지 포맷입니다. Parquet 파일과 트랜잭션 로그(_delta_log)를 결합하여 데이터 웨어하우스 수준의 ACID 보장을 데이터 레이크에서 구현합니다. Databricks 레이크하우스의 모든 테이블은 기본적으로 Delta 포맷을 사용합니다.

## Delta Lake가 제공하는 핵심 보장

<CardGroup cols={2}>
  <Card title="ACID 트랜잭션" icon="shield-check">
    여러 쓰기 작업이 동시에 실행되어도 원자성(Atomicity)이 보장됩니다. 실패한 쓰기는 자동으로 롤백되어 테이블이 일관된 상태를 유지합니다.
  </Card>
  <Card title="스키마 강제 및 진화" icon="table">
    기본적으로 스키마와 맞지 않는 데이터 쓰기를 거부합니다. `mergeSchema` 옵션으로 하위 호환 방식의 스키마 변경을 허용할 수 있습니다.
  </Card>
  <Card title="Time Travel" icon="clock-rotate-left">
    트랜잭션 로그를 기반으로 테이블의 과거 버전을 조회하거나 복원할 수 있습니다. 버전 번호 또는 타임스탬프로 과거 상태에 접근합니다.
  </Card>
  <Card title="DML 전체 지원" icon="pen-to-square">
    `UPDATE`, `DELETE`, `MERGE INTO`를 완전히 지원합니다. 기존 데이터 레이크에서는 불가능했던 CDC(변경 데이터 캡처) 패턴을 구현할 수 있습니다.
  </Card>
</CardGroup>

## Delta 테이블 생성

<Tabs>
  <Tab title="SQL">
    ```sql
    -- 새 Delta 테이블 생성
    CREATE TABLE IF NOT EXISTS sales.silver.customers (
        customer_id   STRING    NOT NULL,
        customer_name STRING,
        email         STRING,
        region        STRING,
        created_at    TIMESTAMP,
        updated_at    TIMESTAMP
    )
    USING DELTA
    PARTITIONED BY (region)
    COMMENT '고객 마스터 테이블 — Silver 레이어'
    TBLPROPERTIES (
        'delta.autoOptimize.optimizeWrite' = 'true',
        'delta.autoOptimize.autoCompact'   = 'true'
    );
    ```
  </Tab>
  <Tab title="PySpark">
    ```python
    from delta.tables import DeltaTable
    from pyspark.sql.types import (
        StructType, StructField, StringType, TimestampType
    )

    schema = StructType([
        StructField("customer_id",   StringType(),    nullable=False),
        StructField("customer_name", StringType()),
        StructField("email",         StringType()),
        StructField("region",        StringType()),
        StructField("created_at",    TimestampType()),
        StructField("updated_at",    TimestampType()),
    ])

    # 빈 Delta 테이블 생성
    (
        DeltaTable.createIfNotExists(spark)
        .tableName("sales.silver.customers")
        .addColumns(schema)
        .partitionedBy("region")
        .property("delta.autoOptimize.optimizeWrite", "true")
        .execute()
    )
    ```
  </Tab>
</Tabs>

## MERGE INTO — 업서트 패턴

`MERGE INTO`는 CDC 처리, SCD Type 1/2, 중복 제거 등에서 가장 자주 사용하는 패턴입니다.

<Tabs>
  <Tab title="SQL MERGE">
    ```sql
    -- 소스: 신규/변경 데이터
    -- 타겟: 기존 고객 테이블
    MERGE INTO sales.silver.customers AS target
    USING (
        SELECT
            customer_id,
            customer_name,
            email,
            region,
            event_time AS updated_at
        FROM bronze.raw_customer_events
        WHERE event_date = current_date()
    ) AS source
    ON target.customer_id = source.customer_id

    -- 일치하는 행: 변경된 컬럼만 업데이트
    WHEN MATCHED AND (
        target.customer_name != source.customer_name
        OR target.email != source.email
    ) THEN UPDATE SET
        target.customer_name = source.customer_name,
        target.email         = source.email,
        target.updated_at    = source.updated_at

    -- 새 행: 삽입
    WHEN NOT MATCHED THEN INSERT (
        customer_id, customer_name, email, region, created_at, updated_at
    ) VALUES (
        source.customer_id,
        source.customer_name,
        source.email,
        source.region,
        current_timestamp(),
        source.updated_at
    );
    ```
  </Tab>
  <Tab title="Python MERGE">
    ```python
    from delta.tables import DeltaTable
    from pyspark.sql.functions import current_timestamp

    target_table = DeltaTable.forName(spark, "sales.silver.customers")

    source_df = (
        spark.table("bronze.raw_customer_events")
        .filter("event_date = current_date()")
        .select("customer_id", "customer_name", "email", "region", "event_time")
    )

    (
        target_table.alias("target")
        .merge(source_df.alias("source"), "target.customer_id = source.customer_id")
        .whenMatchedUpdate(
            condition="target.email != source.email OR target.customer_name != source.customer_name",
            set={
                "customer_name": "source.customer_name",
                "email":         "source.email",
                "updated_at":    "source.event_time",
            },
        )
        .whenNotMatchedInsert(
            values={
                "customer_id":   "source.customer_id",
                "customer_name": "source.customer_name",
                "email":         "source.email",
                "region":        "source.region",
                "created_at":    "current_timestamp()",
                "updated_at":    "source.event_time",
            }
        )
        .execute()
    )
    ```
  </Tab>
</Tabs>

## Time Travel — 과거 버전 조회 및 복원

Delta Lake는 모든 쓰기 작업을 트랜잭션 로그에 기록합니다. 이를 이용해 테이블의 과거 상태를 조회하거나 실수로 삭제한 데이터를 복구할 수 있습니다.

```sql
-- 테이블 변경 이력 확인
DESCRIBE HISTORY sales.silver.customers;

-- 버전 번호로 과거 조회
SELECT * FROM sales.silver.customers VERSION AS OF 5;

-- 타임스탬프로 과거 조회
SELECT * FROM sales.silver.customers
TIMESTAMP AS OF '2024-03-01 09:00:00';

-- 특정 버전과 현재 버전 비교 (추가된 행 확인)
SELECT *
FROM sales.silver.customers VERSION AS OF 10
EXCEPT
SELECT *
FROM sales.silver.customers VERSION AS OF 9;
```

```python
# PySpark에서 Time Travel
df_v5 = spark.read.format("delta") \
    .option("versionAsOf", 5) \
    .table("sales.silver.customers")

df_ts = spark.read.format("delta") \
    .option("timestampAsOf", "2024-03-01") \
    .table("sales.silver.customers")

# 실수로 삭제된 데이터 복원
spark.sql("""
    RESTORE TABLE sales.silver.customers
    TO VERSION AS OF 8
""")
```

<Warning>
  `RESTORE`는 현재 버전을 덮어씁니다. 복원 전에 현재 상태를 별도 테이블에 백업하거나 버전 번호를 기록해 두십시오. `VACUUM` 실행 후에는 해당 버전 이전의 데이터를 복원할 수 없습니다.
</Warning>

## 스키마 강제와 스키마 진화

```python
from pyspark.sql.functions import lit

# 스키마 강제 — 새 컬럼이 있으면 쓰기 실패
try:
    new_df_with_extra_col = customers_df.withColumn("phone", lit("010-0000-0000"))
    new_df_with_extra_col.write.format("delta").mode("append") \
        .saveAsTable("sales.silver.customers")
except Exception as e:
    print(f"스키마 강제로 쓰기 거부됨: {e}")

# 스키마 진화 — 하위 호환 방식으로 새 컬럼 추가 허용
new_df_with_extra_col.write.format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \
    .saveAsTable("sales.silver.customers")

# 또는 DDL로 컬럼 추가
spark.sql("ALTER TABLE sales.silver.customers ADD COLUMN phone STRING")
```

## Z-Ordering으로 쿼리 성능 향상

Z-Ordering은 관련 데이터를 물리적으로 같은 파일에 모아 데이터 스킵 효율을 극대화합니다.

```sql
-- OPTIMIZE + Z-Ordering: 특정 컬럼 기준으로 데이터 재배치
OPTIMIZE sales.silver.orders
ZORDER BY (customer_id, order_date);

-- 실행 결과 확인 (처리된 파일 수, 절감된 크기 등)
-- 출력: numFilesAdded, numFilesRemoved, sizeInBytes 등
```

```python
# Python에서 OPTIMIZE 실행
from delta.tables import DeltaTable

delta_table = DeltaTable.forName(spark, "sales.silver.orders")
delta_table.optimize().executeZOrderBy("customer_id", "order_date")
```

<Info>
  Z-Ordering은 카디널리티가 높고 필터/조인에 자주 사용되는 컬럼에 효과적입니다. 파티션 컬럼에는 Z-Ordering을 적용할 필요가 없습니다 — 파티션 프루닝이 이미 적용되기 때문입니다.
</Info>

## Vacuum으로 불필요한 파일 정리

```sql
-- 7일(기본값) 이전 파일 삭제
VACUUM sales.silver.customers;

-- 보존 기간 변경 (최소 168시간 권장)
VACUUM sales.silver.customers RETAIN 168 HOURS;

-- 삭제될 파일 미리 확인 (DRY RUN)
VACUUM sales.silver.customers RETAIN 168 HOURS DRY RUN;
```

<Warning>
  보존 기간보다 오래된 버전의 Time Travel은 VACUUM 이후 불가능합니다. Time Travel이 필요한 기간을 고려하여 보존 기간을 설정하십시오. 기본값은 7일(168시간)입니다.
</Warning>

## 테이블 속성 관리

<Accordion title="자주 사용하는 테이블 속성">
  ```sql
  ALTER TABLE sales.silver.customers SET TBLPROPERTIES (
      -- 작은 파일 자동 병합 (스트리밍 파이프라인에 유용)
      'delta.autoOptimize.autoCompact'   = 'true',
      -- 쓰기 시 파일 크기 최적화
      'delta.autoOptimize.optimizeWrite' = 'true',
      -- 변경 데이터 피드 활성화 (CDC 소비)
      'delta.enableChangeDataFeed'       = 'true',
      -- 데이터 삭제 벡터 활성화 (삭제 성능 향상)
      'delta.enableDeletionVectors'      = 'true'
  );
  ```
</Accordion>

  -- CDF로 특정 버전 이후 변경분만 조회
