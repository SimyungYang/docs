Unity Catalog는 Databricks의 통합 데이터 거버넌스 솔루션입니다. 여러 워크스페이스에 분산된 데이터 자산(테이블, 파일, ML 모델, 함수)을 하나의 메타스토어에서 중앙 관리하고, 세분화된 접근 제어와 자동 데이터 계보(lineage) 추적을 제공합니다. Unity Catalog 이전에는 각 워크스페이스가 독립된 Hive 메타스토어를 사용했기 때문에 데이터 자산을 일관되게 관리하기 어려웠습니다.

## Unity Catalog가 필요한 이유

<AccordionGroup>
  <Accordion title="기존 Hive 메타스토어의 한계">
    각 Databricks 워크스페이스는 독립된 로컬 Hive 메타스토어를 가집니다. 이 구조에서는 다음과 같은 문제가 발생합니다.

    - **격리된 거버넌스**: 워크스페이스별로 권한을 따로 관리해야 합니다.
    - **데이터 검색 불가**: 다른 워크스페이스의 테이블을 검색하거나 발견할 방법이 없습니다.
    - **계보 추적 부재**: 데이터가 어디서 왔고 어떻게 사용되는지 추적할 수 없습니다.
    - **컬럼 수준 보안 불가**: 테이블 수준보다 세밀한 접근 제어가 어렵습니다.
  </Accordion>
  <Accordion title="Unity Catalog가 해결하는 문제">
    Unity Catalog는 계정 수준 메타스토어를 제공하여 위의 문제를 해결합니다.

    - **중앙화된 거버넌스**: 하나의 메타스토어가 모든 워크스페이스에 적용됩니다.
    - **전사 데이터 검색**: 권한 범위 내의 모든 데이터 자산을 한 곳에서 검색합니다.
    - **자동 계보 추적**: 쿼리·변환·노트북 실행에서 데이터 흐름을 자동으로 기록합니다.
    - **세분화된 접근 제어**: 카탈로그, 스키마, 테이블, 컬럼, 행 수준까지 권한을 설정합니다.
    - **감사 로그**: 누가 언제 어떤 데이터에 접근했는지 모두 기록됩니다.
  </Accordion>
</AccordionGroup>

## 3단계 네임스페이스

Unity Catalog는 `카탈로그.스키마.테이블` 형식의 3단계 네임스페이스를 사용합니다. 이는 기존 Hive의 2단계(`데이터베이스.테이블`)에서 한 단계가 추가된 구조입니다.

```
Unity Catalog 메타스토어
│
├── 카탈로그 (Catalog)          ← 최상위 단위, 환경·도메인별로 분리
│   ├── main                   ← 기본 카탈로그
│   ├── dev_catalog            ← 개발 환경
│   ├── prod_catalog           ← 프로덕션 환경
│   └── marketing_catalog      ← 마케팅 도메인
│
├── 스키마 (Schema/Database)    ← 카탈로그 내 논리적 그룹
│   ├── bronze                 ← 원시 데이터 계층
│   ├── silver                 ← 정제 데이터 계층
│   └── gold                   ← 비즈니스 집계 계층
│
└── 테이블/뷰/함수 (Table)       ← 실제 데이터 자산
    ├── customers              ← 관리형 Delta 테이블
    ├── orders                 ← 관리형 Delta 테이블
    └── daily_sales_vw         ← 뷰
```

### 카탈로그, 스키마, 테이블 생성

```sql
-- 1. 카탈로그 생성
CREATE CATALOG IF NOT EXISTS prod_catalog
COMMENT '프로덕션 데이터 카탈로그';

-- 2. 스키마 생성
CREATE SCHEMA IF NOT EXISTS prod_catalog.gold
COMMENT '비즈니스 집계 데이터 — BI 도구 연결 계층'
MANAGED LOCATION 'abfss://gold@mystorage.dfs.core.windows.net/';

-- 3. 테이블 생성
CREATE TABLE IF NOT EXISTS prod_catalog.gold.monthly_revenue (
  month         DATE         NOT NULL COMMENT '집계 월',
  region        STRING       NOT NULL COMMENT '판매 지역',
  total_revenue DOUBLE       COMMENT '월별 총 매출 (원)',
  order_count   BIGINT       COMMENT '주문 건수',
  updated_at    TIMESTAMP    COMMENT '마지막 갱신 시간'
)
USING DELTA
COMMENT '월별·지역별 매출 집계 테이블'
TBLPROPERTIES (
  'delta.enableChangeDataFeed' = 'true'
);
```

### 기본 카탈로그 설정

```sql
-- 세션의 기본 카탈로그와 스키마를 설정합니다
USE CATALOG prod_catalog;
USE SCHEMA gold;

-- 이제 전체 경로 없이 테이블 이름만으로 쿼리 가능
SELECT * FROM monthly_revenue LIMIT 10;
```

```python
# Python에서 기본 카탈로그·스키마 설정
spark.sql("USE CATALOG prod_catalog")
spark.sql("USE SCHEMA gold")

# 또는 DataFrame API로 전체 경로 지정
df = spark.table("prod_catalog.gold.monthly_revenue")
```

## 권한 관리

Unity Catalog는 ANSI SQL 표준 `GRANT`/`REVOKE` 구문으로 권한을 관리합니다. 권한은 계층 구조로 상속됩니다 — 카탈로그에 부여한 권한은 하위 스키마와 테이블에 자동 적용됩니다.

### 주요 권한 유형

| 권한 | 대상 | 설명 |
|---|---|---|
| `USE CATALOG` | 카탈로그 | 카탈로그에 접근하는 기본 권한 |
| `USE SCHEMA` | 스키마 | 스키마에 접근하는 기본 권한 |
| `SELECT` | 테이블, 뷰 | 데이터 읽기 권한 |
| `MODIFY` | 테이블 | INSERT, UPDATE, DELETE 권한 |
| `CREATE TABLE` | 스키마 | 스키마 내 테이블 생성 권한 |
| `CREATE SCHEMA` | 카탈로그 | 카탈로그 내 스키마 생성 권한 |
| `ALL PRIVILEGES` | 모든 객체 | 해당 객체의 모든 권한 |

### 권한 부여 예시

```sql
-- 분석 팀에 골드 레이어 읽기 권한 부여
GRANT USE CATALOG ON CATALOG prod_catalog TO `analytics-team`;
GRANT USE SCHEMA ON SCHEMA prod_catalog.gold TO `analytics-team`;
GRANT SELECT ON ALL TABLES IN SCHEMA prod_catalog.gold TO `analytics-team`;

-- 특정 테이블에만 읽기 권한 부여
GRANT SELECT ON TABLE prod_catalog.gold.monthly_revenue TO `john.doe@company.com`;

-- 데이터 엔지니어링 팀에 실버 레이어 쓰기 권한 부여
GRANT USE CATALOG ON CATALOG prod_catalog TO `data-engineering`;
GRANT USE SCHEMA ON SCHEMA prod_catalog.silver TO `data-engineering`;
GRANT SELECT, MODIFY ON ALL TABLES IN SCHEMA prod_catalog.silver TO `data-engineering`;
GRANT CREATE TABLE ON SCHEMA prod_catalog.silver TO `data-engineering`;

-- 권한 확인
SHOW GRANTS ON TABLE prod_catalog.gold.monthly_revenue;

-- 권한 취소
REVOKE SELECT ON TABLE prod_catalog.gold.monthly_revenue FROM `john.doe@company.com`;
```

### 행 수준 보안과 컬럼 마스킹

Unity Catalog는 뷰(View)를 통한 행 수준 보안과 동적 데이터 마스킹을 지원합니다.

```sql
-- 행 수준 보안: 사용자가 자신의 리전 데이터만 볼 수 있는 뷰
CREATE VIEW prod_catalog.gold.revenue_by_region_secure AS
SELECT *
FROM prod_catalog.gold.monthly_revenue
WHERE region = current_user_region()  -- 사용자 속성 기반 필터
   OR IS_ACCOUNT_GROUP_MEMBER('data-admin');  -- 관리자는 전체 조회
```

```sql
-- 컬럼 마스킹: 민감 데이터를 마스킹하는 함수 생성
CREATE FUNCTION prod_catalog.security.mask_email(email STRING)
RETURNS STRING
RETURN CASE
  WHEN IS_ACCOUNT_GROUP_MEMBER('data-privacy-team') THEN email
  ELSE CONCAT(LEFT(email, 3), '***@***')
END;

-- 컬럼에 마스킹 함수 적용
ALTER TABLE prod_catalog.silver.customers
ALTER COLUMN email
SET MASK prod_catalog.security.mask_email;
```

<Warning>
  행 수준 보안과 컬럼 마스킹은 Unity Catalog가 활성화된 워크스페이스에서만 작동합니다. Hive 메타스토어 테이블에는 적용되지 않습니다.
</Warning>

## 데이터 계보 (Data Lineage)

Unity Catalog는 쿼리 실행, 노트북, Workflows, Delta Live Tables 작업에서 데이터 계보를 자동으로 추적합니다. 추가 설정 없이 데이터가 어디서 왔고 어디로 흘러가는지 확인할 수 있습니다.

### 계보 확인 방법

1. **Catalog Explorer에서 확인**: **데이터(Data)** → 테이블 선택 → **계보(Lineage)** 탭
2. **REST API로 조회**: Unity Catalog API의 `/api/2.1/unity-catalog/lineage-tracking/table-lineage` 엔드포인트

```python
# Python으로 테이블 계보 조회 (Databricks SDK)
from databricks.sdk import WorkspaceClient

w = WorkspaceClient()

# 업스트림 계보 조회 (이 테이블이 어떤 테이블에서 왔는지)
upstream = w.lineage_tracking.table_lineage(
    table_name="prod_catalog.gold.monthly_revenue",
    include_entity_lineage=True
)

for source in upstream.upstreams:
    print(f"소스 테이블: {source.name}")
    print(f"변환 유형: {source.entity_type}")
```

<Info>
  컬럼 수준 계보(Column-level lineage)도 지원합니다. 특정 컬럼이 어느 소스 컬럼에서 파생됐는지 추적할 수 있어 규정 준수 감사와 영향도 분석에 유용합니다.
</Info>

## 데이터 검색

Unity Catalog Explorer는 권한 범위 내의 모든 데이터 자산을 검색하고 탐색할 수 있는 통합 인터페이스입니다.

### 테이블 문서화

```sql
-- 테이블에 설명 추가
COMMENT ON TABLE prod_catalog.gold.monthly_revenue
IS '월별·지역별 매출 집계 테이블. 매일 오전 2시에 갱신됩니다. 소유팀: 데이터 엔지니어링';

-- 컬럼에 설명 추가
ALTER TABLE prod_catalog.gold.monthly_revenue
ALTER COLUMN total_revenue COMMENT '월별 총 매출액 (단위: 원, VAT 포함)';

ALTER TABLE prod_catalog.gold.monthly_revenue
ALTER COLUMN region COMMENT '판매 지역 코드. 유효값: SEOUL, BUSAN, DAEGU, INCHEON, OTHER';
```

### 태그로 데이터 분류

태그(Tag)를 사용하면 데이터 자산을 분류하고 검색하기 쉽게 만들 수 있습니다.

```sql
-- 테이블에 태그 추가
ALTER TABLE prod_catalog.gold.monthly_revenue
SET TAGS ('domain' = 'finance', 'pii' = 'false', 'sla' = 'gold');

-- 컬럼에 태그 추가 (PII 마킹 등)
ALTER TABLE prod_catalog.silver.customers
ALTER COLUMN phone_number SET TAGS ('pii' = 'true', 'sensitivity' = 'high');

-- 태그로 테이블 검색
SELECT table_catalog, table_schema, table_name, tag_name, tag_value
FROM system.information_schema.table_tags
WHERE tag_name = 'domain' AND tag_value = 'finance';
```

## Unity Catalog 설정 체크리스트

새 워크스페이스에 Unity Catalog를 처음 설정하는 경우 아래 순서를 따르세요.

<Steps>
  <Step title="메타스토어 생성 (계정 관리자)">
    Databricks 계정 콘솔(`accounts.azuredatabricks.net` 또는 클라우드별 URL)에서 메타스토어를 생성합니다. 메타스토어는 클라우드 리전별로 하나를 생성하고 해당 리전의 모든 워크스페이스를 연결합니다.
  </Step>
  <Step title="워크스페이스에 메타스토어 연결">
    계정 콘솔에서 워크스페이스를 선택하고 메타스토어를 연결합니다. 연결 후 워크스페이스의 기본 메타스토어가 Unity Catalog로 변경됩니다.
  </Step>
  <Step title="스토리지 자격증명 설정">
    Unity Catalog가 클라우드 스토리지에 접근할 수 있도록 **스토리지 자격증명(Storage Credential)**을 설정합니다. AWS에서는 IAM Role, Azure에서는 Service Principal, GCP에서는 Service Account를 사용합니다.

    ```sql
    -- Azure: 서비스 주체를 스토리지 자격증명으로 등록
    CREATE STORAGE CREDENTIAL my_storage_cred
    WITH AZURE_SERVICE_PRINCIPAL (
      DIRECTORY_ID = '<tenant-id>',
      APPLICATION_ID = '<client-id>',
      CLIENT_SECRET = '<client-secret>'
    );
    ```
  </Step>
  <Step title="외부 위치 등록">
    스토리지 자격증명을 특정 스토리지 경로와 연결하는 **외부 위치(External Location)**를 등록합니다.

    ```sql
    CREATE EXTERNAL LOCATION prod_data_location
    URL 'abfss://prod@mystorage.dfs.core.windows.net/'
    WITH (STORAGE CREDENTIAL my_storage_cred)
    COMMENT '프로덕션 데이터 스토리지 위치';
    ```
  </Step>
  <Step title="카탈로그와 스키마 구성">
    환경(dev, staging, prod)과 도메인(finance, marketing, operations)을 고려해 카탈로그 구조를 설계하고 생성합니다.
  </Step>
  <Step title="그룹과 권한 설정">
    Databricks 계정 콘솔에서 그룹(Group)을 생성하고, 각 그룹에 역할에 맞는 권한을 부여합니다. 개별 사용자가 아닌 그룹 단위로 권한을 관리하면 유지보수가 쉽습니다.
  </Step>
</Steps>
