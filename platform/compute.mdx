Databricks 컴퓨트(Compute)는 코드와 쿼리를 실행하는 인프라입니다. 워크로드의 특성(대화형 분석, 자동화 파이프라인, SQL 쿼리, ML 학습)에 따라 적합한 컴퓨트 유형을 선택하면 성능을 높이고 비용을 줄일 수 있습니다. 잘못된 컴퓨트 유형을 선택하는 것은 Databricks 비용 초과의 가장 흔한 원인 중 하나입니다.

## 컴퓨트 유형 비교

Databricks는 네 가지 주요 컴퓨트 유형을 제공합니다.

| 유형 | 주요 용도 | 청구 기준 | 인프라 관리 |
|---|---|---|---|
| **All-Purpose 클러스터** | 노트북, 대화형 분석, 개발 | 실행 시간 | 사용자 관리 |
| **Job 클러스터** | 자동화 파이프라인, 스케줄 작업 | 작업 실행 시간 | 자동 생성·삭제 |
| **SQL Warehouse** | SQL 분석, BI 도구 연결 | 쿼리 실행 시간 | 자동 스케일링 |
| **서버리스 컴퓨트** | 노트북, SQL, 작업 | 초 단위 사용량 | Databricks 완전 관리 |

## All-Purpose 클러스터

All-Purpose 클러스터는 여러 사용자가 공유하며 노트북에서 대화형으로 코드를 실행하는 데 사용합니다. 클러스터를 시작하면 사용자가 명시적으로 종료할 때까지 또는 자동 종료 시간이 될 때까지 실행됩니다.

### 언제 사용하나요

- 데이터 탐색 및 시각화
- 노트북 기반 개발·디버깅
- 여러 팀원이 동시에 작업하는 공용 클러스터
- ML 모델 실험 (프로토타이핑 단계)

### 클러스터 생성

<Steps>
  <Step title="컴퓨트 페이지 접속">
    왼쪽 사이드바에서 **컴퓨트(Compute)**를 클릭한 뒤 **클러스터 생성(Create cluster)**을 클릭합니다.
  </Step>
  <Step title="기본 정보 입력">
    ```
    클러스터 이름: team-shared-dev
    정책: Unrestricted (또는 관리자가 설정한 정책)
    클러스터 모드: Multi node (팀 공용) 또는 Single node (개인용)
    ```
  </Step>
  <Step title="Databricks Runtime 선택">
    Databricks Runtime(DBR)은 Spark, Delta Lake, ML 라이브러리를 포함한 실행 환경입니다.

    | Runtime 유형 | 내용 | 권장 상황 |
    |---|---|---|
    | `15.4 LTS` | Spark 3.5, 안정 버전 | 프로덕션 파이프라인 |
    | `15.4 LTS ML` | LTS + MLflow, sklearn, PyTorch | ML 워크로드 |
    | `16.x` | 최신 기능 포함 | 실험·개발 |
    | `GPU Runtime` | CUDA, cuDNN 포함 | 딥러닝, LLM 파인튜닝 |

    <Tip>
      안정적인 파이프라인에는 LTS(Long-Term Support) 버전을 사용하세요. LTS 버전은 2년간 보안 패치와 버그 수정이 지원됩니다.
    </Tip>
  </Step>
  <Step title="노드 유형 및 수 설정">
    ```
    워커 노드 유형: m5.xlarge (AWS) / Standard_DS3_v2 (Azure)
    드라이버 노드 유형: 워커와 동일 (소규모) 또는 더 큰 인스턴스
    워커 수: 2~4 (개발용), 8~20 (프로덕션)
    자동 스케일링: 최소 2, 최대 8 (권장)
    ```
  </Step>
  <Step title="자동 종료 설정">
    **고급 옵션(Advanced Options)** → **자동 종료(Auto termination)**에서 비활성 시간을 설정합니다. 개발 환경에서는 30~60분을 권장합니다.

    <Warning>
      자동 종료를 설정하지 않으면 클러스터가 밤새 실행되어 불필요한 비용이 발생합니다. 반드시 자동 종료를 설정하세요.
    </Warning>
  </Step>
</Steps>

## Job 클러스터

Job 클러스터는 Databricks Workflows에서 작업(job)이 시작될 때 자동으로 생성되고 작업이 완료되면 자동으로 삭제됩니다. 작업 실행 시간에만 비용이 발생하므로 자동화 파이프라인에 가장 경제적입니다.

### 언제 사용하나요

- 야간 배치 ETL 파이프라인
- 스케줄된 ML 모델 재학습
- CI/CD 파이프라인의 데이터 처리 작업
- 정기 보고서 생성

### Workflow에서 Job 클러스터 설정

```json
{
  "job_clusters": [
    {
      "job_cluster_key": "etl_cluster",
      "new_cluster": {
        "spark_version": "15.4.x-scala2.12",
        "node_type_id": "m5.xlarge",
        "num_workers": 4,
        "spark_conf": {
          "spark.databricks.delta.optimizeWrite.enabled": "true",
          "spark.databricks.delta.autoCompact.enabled": "true"
        }
      }
    }
  ],
  "tasks": [
    {
      "task_key": "ingest_data",
      "job_cluster_key": "etl_cluster",
      "notebook_task": {
        "notebook_path": "/Workflows/ingest_raw_data"
      }
    }
  ]
}
```

<Note>
  여러 태스크가 동일한 클러스터를 공유해야 한다면 `job_cluster_key`를 동일하게 설정합니다. 클러스터 재사용으로 시작 시간(약 3~5분)을 줄일 수 있습니다.
</Note>

## SQL Warehouse

SQL Warehouse는 SQL 분석과 BI 도구 연결을 위해 최적화된 컴퓨트 유형입니다. Tableau, Power BI, Looker, dbt 등의 도구를 Databricks에 연결할 때 SQL Warehouse를 사용합니다.

### SQL Warehouse 유형

<Tabs>
  <Tab title="서버리스 SQL Warehouse">
    Databricks가 컴퓨트 인프라를 완전히 관리합니다. 쿼리가 실행될 때 즉시 시작되며(워밍업 없음), 쿼리 완료 후 자동으로 리소스를 반환합니다.

    **장점:**
    - 클러스터 관리 불필요
    - 몇 초 내 시작 (기존 클러스터 대비 10배 빠름)
    - 쿼리 실행 시간에만 비용 발생
    - Databricks가 자동으로 최신 버전으로 업데이트

    **권장 용도:** BI 대시보드, 사용량이 불규칙한 SQL 분석 팀

    ```sql
    -- 서버리스 SQL Warehouse에서 쿼리 실행 예시
    SELECT
      date_trunc('month', sale_date) AS month,
      category,
      SUM(amount) AS monthly_revenue,
      COUNT(DISTINCT customer_id) AS unique_customers
    FROM my_catalog.gold.sales_facts
    WHERE sale_date >= '2025-01-01'
    GROUP BY 1, 2
    ORDER BY 1, 3 DESC;
    ```
  </Tab>
  <Tab title="Pro SQL Warehouse">
    사용자가 클러스터 크기와 자동 스케일링 설정을 제어합니다. 예측 가능한 쿼리 패턴이 있는 팀에 적합합니다.

    **설정 옵션:**
    - **클러스터 크기**: 2X-Small부터 4X-Large까지 (각각 2~128 DBU/시간)
    - **자동 스케일링**: 최소·최대 클러스터 수 설정
    - **자동 일시 중지**: 비활성 후 자동 일시 중지 시간 설정

    ```
    Warehouse 크기 기준 (대략적인 가이드):
    - 2X-Small (2 DBU): 소규모 팀, 개발용
    - Small (4 DBU): 10~20명 분석가 팀
    - Medium (8 DBU): 50명 이상 팀, 복잡한 쿼리
    - Large (16 DBU): 대규모 BI 플랫폼
    ```
  </Tab>
  <Tab title="Classic SQL Warehouse">
    이전 세대의 SQL Warehouse입니다. 서버리스 또는 Pro 유형을 사용할 수 없는 경우에만 선택합니다.

    <Warning>
      신규 워크스페이스에서는 서버리스 또는 Pro SQL Warehouse를 사용하는 것을 권장합니다. Classic은 시작 시간이 길고 기능이 제한됩니다.
    </Warning>
  </Tab>
</Tabs>

### BI 도구 연결

SQL Warehouse에 BI 도구를 연결하려면 **연결 세부 정보(Connection details)** 탭에서 서버 호스트명과 HTTP 경로를 확인합니다.

```
서버 호스트명: adb-1234567890123456.7.azuredatabricks.net
HTTP 경로:    /sql/1.0/warehouses/abc123def456
```

Tableau, Power BI 등 주요 BI 도구는 Databricks 커넥터를 기본 제공합니다. JDBC/ODBC 드라이버로 모든 SQL 도구와 연결할 수 있습니다.

## 서버리스 컴퓨트

서버리스 컴퓨트는 노트북, SQL, Workflows 모두에서 사용할 수 있는 완전 관리형 옵션입니다. 인프라 프로비저닝 없이 즉시 시작되며 초(second) 단위로 청구됩니다.

### 서버리스 지원 범위

| 기능 | 서버리스 지원 여부 |
|---|---|
| 노트북 실행 | ✓ |
| Workflows (Job) | ✓ |
| SQL Warehouse | ✓ |
| Delta Live Tables | ✓ |
| ML 학습 (GPU) | 일부 지원 (확인 필요) |

<Info>
  서버리스 컴퓨트 가용성은 클라우드와 리전에 따라 다를 수 있습니다. Databricks 계정 페이지에서 지원 여부를 확인하세요.
</Info>

## 비용 최적화

컴퓨트 비용은 Databricks 사용 비용의 대부분을 차지합니다. 아래 전략을 적용하면 비용을 크게 줄일 수 있습니다.

<AccordionGroup>
  <Accordion title="적절한 클러스터 크기 선택">
    클러스터는 크다고 항상 빠른 것이 아닙니다. 네트워크 셔플이 많은 작업에서는 워커 수를 늘려도 성능이 선형으로 향상되지 않습니다.

    **권장 방법:**
    1. 소규모 클러스터(2~4 워커)로 시작해 Spark UI에서 병목 지점을 확인합니다.
    2. 특정 스테이지가 오래 걸린다면 워커 수를 늘립니다.
    3. 메모리 집약적 작업(ML, 대규모 조인)에는 메모리 최적화 인스턴스(`r5`, `Standard_E`)를 사용합니다.
    4. CPU 집약적 작업에는 컴퓨트 최적화 인스턴스(`c5`, `Standard_F`)를 사용합니다.
  </Accordion>
  <Accordion title="자동 스케일링 활성화">
    자동 스케일링을 사용하면 워크로드에 따라 클러스터 크기가 자동으로 조정됩니다.

    ```python
    # Databricks API로 자동 스케일링 설정 예시
    cluster_config = {
        "autoscale": {
            "min_workers": 2,
            "max_workers": 10
        },
        "spark_version": "15.4.x-scala2.12",
        "node_type_id": "m5.xlarge"
    }
    ```

    <Note>
      Spark Structured Streaming 작업에는 자동 스케일링이 적합하지 않을 수 있습니다. 스트리밍 작업에서는 고정된 워커 수를 사용하는 것이 더 예측 가능합니다.
    </Note>
  </Accordion>
  <Accordion title="클러스터 정책으로 비용 통제">
    클러스터 정책(Cluster Policy)을 사용하면 사용자가 생성할 수 있는 클러스터 유형과 크기를 제한할 수 있습니다.

    ```json
    {
      "node_type_id": {
        "type": "allowlist",
        "values": ["m5.xlarge", "m5.2xlarge"],
        "defaultValue": "m5.xlarge"
      },
      "autotermination_minutes": {
        "type": "range",
        "minValue": 10,
        "maxValue": 120,
        "defaultValue": 30
      },
      "num_workers": {
        "type": "range",
        "minValue": 1,
        "maxValue": 8
      }
    }
    ```
  </Accordion>
  <Accordion title="스팟/프리엠프터블 인스턴스 활용">
    배치 작업과 내결함성(fault-tolerant) 파이프라인에는 스팟 인스턴스(AWS Spot, Azure Spot, GCP Preemptible)를 사용하면 비용을 50~70% 절감할 수 있습니다.

    드라이버 노드는 온디맨드 인스턴스를 사용하고 워커 노드에만 스팟 인스턴스를 적용하는 것을 권장합니다. 스팟 인스턴스 중단 시 Databricks는 자동으로 태스크를 재시도합니다.

    <Warning>
      스팟 인스턴스는 클라우드 공급자에 의해 갑자기 회수될 수 있습니다. 실시간 서비스나 SLA가 엄격한 작업에는 온디맨드 인스턴스를 사용하세요.
    </Warning>
  </Accordion>
  <Accordion title="Photon 엔진으로 성능 향상">
    Photon은 Databricks가 개발한 네이티브 벡터화 쿼리 엔진으로, SQL과 DataFrame API 작업을 최대 12배 빠르게 실행합니다. 추가 설정 없이 Photon을 지원하는 인스턴스를 선택하면 자동으로 활성화됩니다.

    Photon은 특히 집계 쿼리, 조인, 정렬, 스캔 작업에서 큰 성능 향상을 보입니다.
  </Accordion>
</AccordionGroup>

## 클러스터 모니터링

실행 중인 클러스터의 성능을 모니터링하려면 클러스터 상세 페이지의 **Spark UI** 탭을 사용합니다.

```python
# 클러스터 성능 문제 진단에 유용한 Spark 설정
spark.conf.set("spark.sql.adaptive.enabled", "true")  # 적응형 쿼리 실행
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")  # 파티션 자동 병합
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")  # 데이터 스큐 자동 처리

# 현재 설정 확인
print(spark.conf.get("spark.sql.shuffle.partitions"))
```

<Tip>
  Spark UI의 **Stages** 탭에서 오래 걸리는 스테이지를 찾고, **Tasks** 탭에서 데이터 스큐(특정 태스크만 오래 걸리는 현상)를 확인하세요. AQE(Adaptive Query Execution)를 활성화하면 스큐와 파티션 문제를 자동으로 처리합니다.
</Tip>
